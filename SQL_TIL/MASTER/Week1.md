# 1️⃣ 개념 정리

## 01. 여러 가지 과제 

<!-- 본인의 희망 직무와 관련하여 어떤 과제가 있을지 생각해보고, 느낀 바를 적어주세요. -->

소규모 기업의 경우 엑셀 수준에서 원장 및 거래 데이터 분석이 가능하다.<BR>
그러나 종업원 및 거래규모, 자산규모가 큰 기업이라던가, 금융기업처럼 자동적으로 기록되는 거래가 무수히 많은 경우 엑셀의 한계점에 직면하게 된다.<br>
우선 약 110만행의 한도 때문에 하나의 탭에 데이터를 모두 모을 수 없다.<br>
그 결과 전체 데이터를 가지고 샘플을 추출하는 MUS를 수행하지 못할 뿐더러, 추세를 확인하는 분석적 절차에도 차질이 생긴다.<br>
또한, 지나치게 용량이 큰 엑셀 파일로 인해 컴퓨터가 먹통이 되는 일을 쉽게 겪을 수 있다.

이 경우 빅데이터 분석 능력이 요구된다.<br>
특정 유형의 거래의 추세를 분석하여 비경상적인 상황을 식별하고 이를 담당자에게 질의하여 추가적인 증빙을 확인해야 한다.<br>
IT전문가가 회계법인 내에 존재하지만 감사 업무를 수행하는 팀원들은 데이터 분석 관련 지식을 가지고 있지 않아 상호간의 의견 충돌이 발생한다.<br>
가장 우선적으론 투입되는 자원에 대한 이해가 다르며, 제공되어야 하는 데이터의 형태를 이해하지 못한다.<br>
빅데이터 분석 능력을 길러 IT 스페셜리스트와 원활히 소통하고, 더 나아가 감사업무팀 내에서 적정 수준의 분석 절차를 수행하는 능력을 기르는 것이 이번 학기의 목표이다.


## 02. 시스템

<!-- 교재에서 나열된 도구들의 특징을 정리해주세요. -->

- PostgreSQL
    - 오픈소스 RDB
    - GUI 인스틀로 제공
    - 표준 SQL 준수
    - 확장 기능을 충분히 제공
    - 로컬에서 가볍게 실행됨
- Apache Hive
    - 분산 파일 시스템(HDFS): 거대 데이터를 작게 분할하여 각 디스크에서 동시에 읽어 들여 고속으로 대량의 데이터 처리
    - 파일 기반의 시스템
    - 쿼리 실행 시 동적으로 데이터 정의 가능
    - UDF가 풍부함
- Amazon Redshift
    - AWS에서 제공하는 분산 병렬 RDB
    - 대량의 데이터와 상호 작용하는 쿼리를 실행할 때 효과적
    - 클라우드 서비스
    - 사용 시간에 따라 비용이 발생함
- Google BigQuery
    - 구글이 제공하는 클라우스 서비스
    - 직접 노드 인스턴스를 관리할 필요가 없음
    - 읽어 들인 데이터의 양으로 비용이 발생함
    - 구글이 제공하는 다른 클라우드 서비스와 쉽게 연동할 수 있음
- SparkSQL
    - 분산 처리 프레임워크인 Apache Spark의 기능 중 SQL 인터페이스와 관련된 기능
    - 오픈 소스 프레임워크
    - 가장 빠른 속도로 개발이 이루어짐
    - 빅데이터 활용과 관련된 대부분의 처리를 한 번에 구현할 수 있음




## 03. 데이터 
```
✅ 학습 목차
* 데이터의 종류
* 업무 데이터
* 로그 데이터
* 두 데이터를 사용해서 생성되는 가치
```

<!-- 이 부분을 지우고 새롭게 배운 내용을 자유롭게 정리해주세요. -->

**데이터의 분류**
- 업무 데이터: 서비스와 시스템을 운용하기 위한 목적으로 구축된 데이터베이스에 존재하는 데이터
    - 트랜잭션 데이터: 서비스와 시스템을 통해 사용자의 행동을 기록한 데이터
    - 마스터 데이터: 서비스와 시스템이 정의하고 있는 데이터
- 로그 데이터
    - 통계 또는 분석을 주 용도로 설계된 데이터
    - 특정 태그를 포함하여 전송된 데이터
    - 특정 행동을 서버 측에 출력한 데이터

<br>
<br>

**업무 데이터**

- 업무 데이터의 특징
    - 높은 데이터 정밀도
    - 갱신형 데이터
    - 다량의 테이블 수
- 업무 데이터 축적 방법
    - 모든 데이터 변경
    - 모든 레코드의 스냅샷을 날짜별로 저장
    - 어제와의 변경 사항만 누적
- 업무 데이터 활용시 유의사항
    - 매출액, 사용자 수처럼 정확한 값을 요구할 경우 활용
    - 서비스의 방문 횟수, 페이지뷰, 사용자 유도 등의 데이터 분석에는 사용할 수 없음
    - 데이터 변경이 발생할 수 있으므로 추출 시점에 따라 결과가 변화할 수 있음

<br>
<br>

**로그 데이터**

- 로그 데이터의 특징
    - 시간, 사용자 엔드 포인트, IP, URL, 레퍼러, Cookie 등의 정보 저장
    - 추출 방법에 따라 데이터의 정밀도가 달라짐
    - 계속 기록을 추가하는 것뿐이므로 과거 데이터가 변경되지 않음
- 로그 데이터 축적 방법
    - 태그, SDK를 통해 사용자 장치에서 데이터를 전송하고 출력하기(비컨 형태)
    - 서버에서 데이터를 추출하고 출력하기(서버 형태)
- 로그 데이터 다루기
    - 사이트 방문 횟수, 페이지뷰, 사용자 유도 상황을 집계하고 분석할 때 주로 사용
    - 최신 상태를 고려한 분석에는 적합하지 않음
    - 계속 기록을 누적하는 형태이므로 추출 결과가 변할 가능성이 적음
    - 데이터의 정확도는 업무 데이터에 비해 낮음
